# ブログ執筆スタイルガイド

**クイックリファレンス**: このガイドは、hiroppy/site ブログ記事の文体、トーン、構成の規約をまとめたものです。新しいブログ記事を書く際に、一貫性を保つために使用してください。

2017-2025年の16以上の記事を分析して作成しました。

---

## 1. 重要情報（必読）

### ファイル設定

- **保存場所**: `/src/content/blog/*.mdx`
- **言語**: 日本語（主）、技術用語は英語
- **視点**: 一人称（自分、私）

### 必須のFrontmatter

```yaml
---
title: "日本語のタイトル"
date: YYYY-MM-DD
description: "1-2文の簡潔な説明"
image: /images/[path/to/ogp]
tags: [tag1, tag2, tag3] # カンマ区切り、スペースなし
---
```

### よく使うMDXインポート

```tsx
import { Image } from "../../components/Image";
import { TwitterCard } from "../../mdx/components/TwitterCard";
import { YoutubeCard } from "../../mdx/components/YoutubeCard";
```

**注**: OGコンポーネントは不要です。`[リンクテキスト](url)` で自動的にリンクプレビューが展開されます。

## 2. トーンと声

### 全体的な特徴

- **会話的でありながらプロフェッショナル**: 同僚とコーヒーを飲みながら話すような感じ
- **謙虚で内省的**: 自慢的な言葉を避け、限界を認める
- **一人称視点**: 「自分」や「私」を自由に使う
- **不確実性について正直**: 分からないことを表現する

### 声のパターン

**感謝**: "感謝しています", "ありがとうございました", "〜のおかげで"
**謙遜**: "微力ながら", "案外", "少しだけ"
**自己認識**: "自分にとって", "正直", "個人的には"
**不確実性**: "かなと思う", "気がする", "のではないかと思っています"

### よく使う接続詞・転換フレーズ

**冒頭**:

- "さて、"
- "今回は、" / "今回、"
- "最近、"

**説明**:

- "つまり、"
- "というのも、"
- "そこで、"

**情報追加**:

- "また、"
- "ちなみに、"
- "加えて、"

**対比**:

- "ただ、" / "ただし、"
- "しかしながら、"
- "一方で、"

**結果・結論**:

- "なので、"
- "結果的に、"
- "だからこそ、"

## 3. 記事構成テンプレート

### テンプレート1: 個人的/内省的な記事

**例**: `30-years-old.mdx`, `join-yuimedi.mdx`, `2021.mdx`, `2022.mdx`, `2023.mdx`

**構成**:

```markdown
[冒頭段落: 文脈と個人的な状況を設定]

## [主要テーマ1] (例: "仕事", "OSS", "VPoE")

[具体例を含む内省的な叙述]
[過去記事タイトル](url) [過去記事への自己参照]

## [主要テーマ2] (例: "生活", "なぜやめるのか?")

[内省の続き]

## [主要テーマ3] (オプション)

[追加のテーマ]

### [サブセクション] (控えめに使用)

[特定のトピックへの深掘り]

## さいごに / 今後

[今後の展望または最終的な考え]
```

**特徴**:

- 叙事的、時系列の流れ
- `[リンクテキスト](url)` による自己参照を多用（自動的にリンクプレビュー展開）
- 具体的な日付、会社名、同僚への言及
- コード比率: 0-10%
- 長さ: 3,000-5,000文字
- 常に同僚/会社への感謝を含む
- 個人的な目標についての前向きな発言

**例（冒頭）**:

> "30歳という節目でもあり、社会人になって、7年経ったのでせっかくなので振り返ろうかと思う。"

### テンプレート2: 技術的な深掘り記事

**例**: `nodejs-strip-type.mdx`, `webpack-persistent-caching.mdx`, `nodejs-event-loop.mdx`

**構成**:

````markdown
[関連リンクタイトル](url) [関連するissue/PR/ドキュメントへのリンク]

[オプションの免責事項: 💁‍♀️ まだマージされてない点に注意してください]

[冒頭: この記事が何をカバーするかの簡単な文脈]

## 背景

[この技術/機能がなぜ存在するかを説明]

## [機能/概念の名前]

[詳細な技術的説明]

### [特定の詳細のためのサブセクション]

[コード例、図、説明]

```javascript
// 注釈付きのコード例
```
````

## 仕組み / どのように動作するか?

[内部実装の詳細]

## 注意点 / 制限 / Caveats

[制限事項、落とし穴、エッジケースのリスト]

## まとめ

- 重要なポイントの箇条書きサマリー
- 3-5の簡潔なポイント

`````

**特徴**:
- コード比率: 30-50%
- 重要でないコードには `::: details` を使用、複数コードの場合は `::: code-group` を使用
- 公式ドキュメント/issue への `[リンクテキスト](url)` 参照を頻繁に使用（自動展開）
- 明確なセクション階層 (##, ###, ####)
- 実験的な機能には冒頭に謙虚な免責事項
- 長さ: 8,000-22,000文字

**コード例のパターン**:
````markdown
::: details lib/internal/modules/cjs/loader.js

```javascript
// 説明的なコメント付きの長いコード（重要度が低い場合）
`````

:::

`````

### テンプレート3: チュートリアル/ハウツー記事
**例**: `nextjs-app-router-training.mdx`, `create-feedle.mdx`, `npm-variable.mdx`

**構成**:
```markdown

## 目的
- 目標/動機のリスト
- 2-4の箇条書き

## 手順 / 実装 / 知見
[段階的な手順または実装の詳細]

### [ステップ1]
[説明付きのコード例]

### [ステップ2]

## [追加セクション]
- FAQ
- 注意点
- トラブルシューティング

## さいごに / まとめ
[学びと振り返り]
```

**特徴**:
- 冒頭に明確な目標声明
- コード比率: 20-40%
- 実用的で実行可能な例
- リポジトリ/デモへのリンク
- 学びについての締めくくりの考え
- 長さ: 10,000-18,000文字

### テンプレート4: イベント/体験レポート

**特徴**:
- リッチメディア: 画像、TwitterCard、YoutubeCard
- 地理的/文化的文脈
- 個人的な叙述 + 事実情報
- コード比率: 0-10%
- ビジュアル重視のセクション
- 長さ: 3,500-9,000文字

## 4. フォーマット規約

### 見出し
- **タイトル**: H1（frontmatter経由のみ、本文には使用しない）
- **主要セクション**: `## セクション名`
- **サブセクション**: `### サブセクション`（控えめに使用）
- **見出しレベルをスキップしない**（## → #### は不可）

### コードブロック
常に言語を指定:
```javascript
// 説明的な変数名を使用
// 必要に応じて日本語または英語でコメント
```

**複数パッケージマネージャーの例**（VitePressスタイル）:
````markdown
::: code-group

```bash [npm]
npm install package-name
```

```bash [pnpm]
pnpm add package-name
```

```bash [yarn]
yarn add package-name
```

:::
`````

**長いコードスニペット**（`::: details` を使用）:

````markdown
::: details 説明的なファイル名またはタイトル

```js
// 長いコード（重要度が低い場合）
```

:::
````

**注**: 重要でないコードには `::: details` を使用。複数のコード例がある場合は `::: code-group` を使用します。

### リスト

- 順序なしリストには `-` を使用（`*` や `+` ではなく）
- 読みやすさのためにリスト項目間に空行を追加（オプション）
- ネストされたリスト: 2スペースでインデント

### リンクと参照

- **リンクプレビュー**: `[リンクテキスト](url)` で自動的にリンクプレビューが展開されます
- **自己参照**: `[過去記事のタイトル](https://blog.hiroppy.me/entry/...)`
- **外部参照**: `[参照先のタイトル](url)`
- **Twitter**: `<TwitterCard id="..." height={420} />`（要インポート）
- **YouTube**: `<YoutubeCard id="..." />`（要インポート）

### 強調

- **太字**: `**テキスト**` で重要な用語、ラベル、キー概念
- _斜体_: ほとんど使用しない
- **インラインコード**: `` `code` `` で技術用語、コマンド、ファイル名、変数名

### 免責事項と注意書き

記事の冒頭で絵文字や特別なマーカーを使用:

```markdown
💁‍♀️ まだマージされてない点に注意してください

注意: これはまだ未定です。

注意: これは公式見解ではない、ただ 1 メンバーの感想文です。
```

### スペーシング

- **空行**: 明示的なスペースが必要な場合は `<br />` を追加
- **セクション間**: `##` 見出しの前に1行の空行
- **コードブロック周辺**: 前後に空行

## 5. カテゴリ別ガイドライン

### 技術記事

**タグ**: `node.js`, `next.js`, `webpack`, `javascript`, `typescript`

- **コード比率**: 30-50%
- **構成**: 背景 → 説明 → 仕組み → 注意点 → まとめ
- **トーン**: 教育的、詳細、限界について謙虚
- **参照**: 公式ドキュメント、issue、PRへの `[リンクテキスト](url)` を多用（自動展開）
- **例**: 常に実行可能なコードスニペットを含む
- **免責事項**: 実験的な機能には警告を追加（💁‍♀️）

### 個人的/キャリア記事

**タグ**: `life`, `job`, `ポエム`

- **コード比率**: 0-10%
- **構成**: 叙事的、時系列
- **トーン**: 内省的、感謝的、動機について正直
- **参照**: 過去記事への自己参照 `[記事タイトル](url)`（自動展開）
- **内容**: 具体的な日付、会社名、同僚への言及
- **締めくくり**: 常に今後の展望を含む（"今後"、"これから"）

### OSS/プロジェクト更新

**タグ**: `oss`, `webpack`, `github`

- **コード比率**: 20-40%
- **構成**: 文脈 → 作ったもの → 学び → まとめ
- **トーン**: 成果について謙虚、コミュニティ/協力に焦点
- **参照**: リポジトリ、デモ、関連プロジェクトへのリンク
- **帰属**: 協力者、インスピレーションを明示的にクレジット

### サイト/メタ記事

**タグ**: `site`, `astro`, `next.js`

- **コード比率**: 10-30%
- **構成**: 何が変わったか → なぜ → どのように → 結果
- **トーン**: 技術的だがアクセスしやすい
- **内容**: 移行ストーリー、ツール選択、技術的決定

## 7. ユニークな要素

### 自己参照リンク

関連する場合は常に過去の記事にリンク:

```markdown
[2018年の振り返り](https://blog.hiroppy.me/entry/2018)
```

`[リンクテキスト](url)` 形式で書くと、自動的にリンクプレビューが展開され、記事間の叙事的な連続性が生まれます。

**一般的なパターン**:

- 新しい年次レビューを書く際に過去の年次レビューを参照
- 関連する技術記事へのリンク
- キャリアの変遷を時系列で接続

### コミュニティへの帰属

協力者、OSS貢献者、同僚をクレジット:

- "〜をはじめ〜"
- "〜のおかげで〜"
- "感謝しています"
- 人々のTwitter/GitHubプロフィールにリンク
- 貢献者を明示的に名前で言及

**例**:

> "[@koba](https://twitter.com/koba04)さんと[@yoshiko](https://twitter.com/yoshiko_pg)さんで〜"

### 時間的マーカー

具体的な日付と期間を含める:

- "3/2 - 8/26 までアイルランドで生活していました。"
- "2019年11月に二回目の正式な入社をしました。"
- "今年の2/20に籍を入れて〜"
- "2021年は〜"
- "最近は〜"

### Hatena移行の文脈

多くの記事はfrontmatterに `hatenaPath` があります（Hatena Blogから移行）:

```yaml
hatenaPath: entry/2018/12/03/090000
```

これらは元のスタイルと構造を維持しています。

## 重要な執筆原則

1. **ジャーゴンではなく文脈から始める** - 「どのように」に入る前に常に「なぜ」を説明
2. **戦略的に一人称を使う** - 個人的な経験は信頼性を追加
3. **視覚的な区切りが重要** - 見出し、リスト、コードブロックで文章の壁を防ぐ
4. **自由にリンクする** - 過去の作品や外部の権威を参照
5. **謙虚さを示す** - 知らないことや間違っているかもしれないことを認める
6. **技術的精度と会話的な流れを混ぜる** - 正確さと読みやすさのバランス
7. **スキャン可能な構造** - 読者は見出し/リストから主要なアイデアを把握できるべき
8. **具体例を提供** - 抽象的な概念にはコード、図、または具体的な例が必要
9. **前進で終わる** - 結論は将来の作業や応用を指す
10. **コミュニティを埋め込む** - 貢献者をクレジット、議論にリンク、インスピレーションを認める
